MVC: mapping and dispatching of client requests.

The Iglu model-view-control package offers a mechanism to map requests to module methods in the application.

Somehow, especially from a web environment (that was intended to contain documents instead of applications) it has always been difficult to interact with server-side applications.
The essence of a good system that facilitates the interaction of user interface and server-side application, is to keep the structure as compact as possible. Iglu MVC defines a minimal number of parts that need maintenance:

  • A user interface part, typically a JSP that accesses the application via service components and agents and that has access to empty or (partially) filled out forms.
  • A mapping part, a readable plain text file, that defines incoming requests, the module methods they access and the user interface part they get directed to depending on the outcome of the methods invoked.
  • A controller part, being an agent or service component, that processes request data and returns clues to what to next.
In case of a web application, requests are collected and dispatched by a servlet that implements RequestDispatcher. The DispatcherServlet delegates requests to a web-independent RequestMapper, which is configured as an Iglu component. The RequestMapper manages one or more mapping structures that are based on a map configuration. These structures may invoke methods on modules. After a process is finished they delegate dispatching to the RequestDispatcher again. In terms of the classical model-view-control principle:
  • The model is (represented by) the service layer of the application in the form of services and stateful agents.
  • The view is represented by JSP-files.
  • The controller is the request mapper - possibly in combination with modules in the presentation layer - that decides, which business logic to invoke and which view to dispatch to.
